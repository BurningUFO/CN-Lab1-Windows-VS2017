# 5.1 实际上机调试时间

结合本次实验的分工和仓库中的开发材料，我估计自己完成本次实验所投入的实际上机调试时间约为 **7 小时**。这个时间比“如果一切顺利，4~6 小时即可完成”的理想预测略长一些，但总体仍然比较接近正常范围。

之所以会超过预测，并不是因为开发环境不熟悉。我此前已经使用过 Visual Studio 2022，因此在 Windows 环境下打开工程、确认编译配置、完成基本编译和启动 A/B 站程序这一部分花费的时间相对较少。真正耗时的部分，主要集中在选择重传协议的核心实现、双站联调和问题定位上。

按照我在小组中的分工，我主要负责 `datalink.c` 中协议核心逻辑的实现，包括发送窗口、接收窗口、失序缓存、ACK/NAK 机制以及超时重传等内容。因此，我的时间投入大致可以分为以下几个部分：

| 阶段 | 主要内容 | 估计耗时 |
| --- | --- | ---: |
| 实验框架阅读与方案确定 | 阅读实验框架，理解 `protocol.c` / `protocol.h` 提供的事件与接口，确定采用选择重传协议并梳理状态变量与帧格式 | 1.5 小时 |
| 工程编译与基础运行确认 | 使用 Visual Studio 2022 打开工程，确认编译配置，完成基础编译和 A/B 站程序启动验证 | 0.5 小时 |
| 协议核心代码实现 | 完成发送窗口、接收窗口、缓存、ACK/NAK、定时器和重传逻辑等主体代码 | 2.5 小时 |
| 双站联调与错误修复 | 根据运行日志定位窗口推进、确认处理、乱序缓存、重复重传等问题，并反复修改代码 | 1.5 小时 |
| 参数调整与稳定性复测 | 配合测试结果，对定时器和发送队列相关参数做调整，并验证程序是否稳定运行 | 1 小时 |
| 合计 |  | 7 小时 |

本次实验耗时略高于理想预测，主要原因有三点。第一，虽然我对 Visual Studio 2022 比较熟悉，因此工程配置和编译环境本身没有耗费太多时间，但协议逻辑的正确性仍然需要反复验证。很多问题不是编译时报错，而是程序能够运行，但在日志中会表现出确认处理不够理想、窗口推进不顺畅或者超时重传偏多等现象。第二，选择重传协议同时涉及发送端和接收端状态维护，尤其是序号管理、缓存处理、ACK/NAK 配合和定时器控制，逻辑上比普通 C 语言程序更复杂。第三，实验不仅要保证程序能跑起来，还要保证双站联调后能够比较稳定地工作，因此还需要额外花时间做短时复测和简单参数调整。

总体来看，由于我对开发环境已经比较熟悉，所以这次并没有在安装和工具使用上花费太多时间；实际耗时主要还是集中在协议实现、双站联调和问题修正上。因此，将本次实验的实际上机调试时间写为 **约 7 小时**，更符合我的实际情况。
