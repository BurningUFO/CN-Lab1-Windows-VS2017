# 实验一：选择重传实验指导（精简版）

本文件由 `计算机网络实验一.pdf` 提取并人工裁剪而成，只保留对“选择重传协议”实验实现、运行、调试、测试有直接指导作用的内容。

## 1. 实验目标与硬性约束

- 在仿真环境下实现**有噪声信道环境下两站点之间的无差错全双工通信**。
- 本组已确定实现**选择重传协议**。
- 信道模型固定如下：
  - `8000 bps` 全双工卫星信道
  - 传播时延 `270 ms`
  - 误码率 `1e-5`
  - 信道提供**字节流传输服务**
  - 网络层分组长度固定为 `256` 字节
- 需要根据实际信道情况合理设置协议工作参数，重点包括：
  - 滑动窗口大小
  - 数据帧重传定时器时限
  - ACK 搭载定时器时限
- 除了保证无差错传输，还需要尽量提高线路利用率。

### 2.2 协议设计和总体设计

- 设计选择重传协议的数据帧字段和控制逻辑。
- 规划程序总体结构及相关子程序。

### 2.3 编码和调试

- 在**无误码信道**下测试并记录线路利用率。

### 3.2 运行模型

- 只生成一个可执行文件：`datalink.exe`
- 在两个命令行窗口中分别启动站点 `A` 和 `B`
- 站点角色：
  - `A`：TCP 服务器，默认监听端口 `59144`
  - `B`：TCP 客户端，默认连接 `127.0.0.1:59144`
- 如果端口被占用，可通过命令行参数改端口。
- Windows 防火墙需要允许相关 TCP 端口访问。

启动示例：

```text
datalink -d3 A
datalink -d3 B
```

### 3.3 仿真环境结构

程序由三层组成：

- 物理层：实验库提供
- 数据链路层：你自行实现
- 网络层：实验库提供，仅用于产生和校验固定模式分组流

仿真环境的关键事实：

- 网络层分组长度固定为 `256` 字节
- 物理层发送与接收采用帧接口 `send_frame()` / `recv_frame()`
- 实验库已经处理了信道速率、误码注入和传播时延
- 为简化实验，物理层**不负责成帧**，链路层直接按“帧”单位收发

### 3.4 正确性验证方式

- A 端和 B 端都会使用固定模式的“分组序列发生器”。
- 接收方按同样公式检查收到分组的内容和顺序。
- 一旦链路层出错，程序会打印错误并中止。
- 每个分组前两个字节可视为分组 ID，便于调试重传行为：
  - A 端产生的分组 ID 范围：`10000~19999`
  - B 端产生的分组 ID 范围：`20000~29999`

## 4. 协议库接口

### 4.1 初始化

`main()` 必须首先调用：

```c
void protocol_init(int argc, char **argv);
```

作用：

- 建立 A/B 两站之间的 TCP 连接
- 初始化运行参数
- 设置统一的时间坐标零点
- 从命令行读取调试、端口、误码率、日志文件等选项

### 4.2 网络层接口

```c
#define PKT_LEN 256

void enable_network_layer(void);
void disable_network_layer(void);
int get_packet(unsigned char *packet);
void put_packet(unsigned char *packet, int len);
```

说明：

- `enable_network_layer()` / `disable_network_layer()` 用于控制网络层是否继续交付新分组。
- 只有在发生 `NETWORK_LAYER_READY` 事件后，才能调用 `get_packet()` 获取下一个待发分组。
- 收到正确分组后调用 `put_packet()` 上交网络层。
- `put_packet()` 内部带统计功能；若接收持续正常，接收端会周期性输出类似报告：

```text
480.484 .... 1784 packets received, 7611 bps, 95.14%, Err 38 (9.9e-006)
```

这条报告可直接用于判断：

- 协议是否持续正常工作
- 实际有效吞吐率
- 实际线路利用率
- 检出的帧差错数量与估计误码率

如果长时间不再出现该报告，通常说明协议进入死锁或失效状态。

### 4.3 事件驱动接口

```c
int wait_for_event(int *arg);

#define NETWORK_LAYER_READY  0
#define PHYSICAL_LAYER_READY 1
#define FRAME_RECEIVED       2
#define DATA_TIMEOUT         3
#define ACK_TIMEOUT          4
```

事件含义：

- `NETWORK_LAYER_READY`：网络层有新分组待发，此时才允许 `get_packet()`
- `PHYSICAL_LAYER_READY`：物理层发送队列长度降到 `50` 字节以下
- `FRAME_RECEIVED`：收到一整帧，此时才允许 `recv_frame()`
- `DATA_TIMEOUT`：某个数据定时器超时，`arg` 返回定时器编号
- `ACK_TIMEOUT`：ACK 搭载定时器超时

主循环框架可按如下形式组织：

```c
enable_network_layer();
for (;;) {
    event = wait_for_event(&arg);
    switch (event) {
    case NETWORK_LAYER_READY:
        len = get_packet(my_buf);
        ...
        break;
    case PHYSICAL_LAYER_READY:
        ...
        break;
    case FRAME_RECEIVED:
        rbuf_len = recv_frame(rbuf, sizeof rbuf);
        ...
        break;
    case ACK_TIMEOUT:
        ...
        break;
    case DATA_TIMEOUT:
        ...
        break;
    }

    if (...)
        enable_network_layer();
    else
        disable_network_layer();
}
```

### 4.4 物理层接口

```c
void send_frame(unsigned char *frame, int len);
int recv_frame(unsigned char *buf, int size);
int phl_sq_len(void);
```

说明：

- `send_frame()`：发送一帧。每字节发送需 `1 ms`，帧与帧之间保留 `1 ms` 边界时间。
- `recv_frame()`：接收一帧，返回实际长度。
- `phl_sq_len()`：获取当前物理层发送队列长度。

必须注意：

- 物理层发送队列最多可缓存 `64 KB`。
- 在一个事件处理周期内只要调用过一次 `send_frame()`，之后 `wait_for_event()` 会在发送队列降到 `50` 字节以下时产生 `PHYSICAL_LAYER_READY`。
- 如果在 `PHYSICAL_LAYER_READY` 到来时暂时没数据发，程序需要自行记录“物理层已准备好”状态；因为系统不会重复发这个事件。
- 不顾物理层状态连续发送多帧，会导致大量数据堆积在物理层发送队列，拖慢超时与重传行为。

## 5. CRC、定时器与调试

### 5.1 CRC-32

```c
unsigned int crc32(unsigned char *buf, int len);
```

要求：

- 本实验统一使用 `CRC-32`
- 生成多项式与 IEEE 802.3 以太网相同

附加 CRC 的典型写法：

```c
*(unsigned int *)(p + 243) = crc32(p, 243);
```

校验方法：

```c
crc32(p, 243 + 4) == 0
```

注意：

- 缓冲区必须预留足够空间，避免越界。
- 收到帧后应先做 CRC 检查，错误帧不能交付上层。

### 5.2 定时器

```c
unsigned int get_ms(void);
void start_timer(unsigned int nr, unsigned int ms);
void stop_timer(unsigned int nr);
void start_ack_timer(unsigned int ms);
void stop_ack_timer(void);
```

关键行为：

- `start_timer(nr, ms)`：启动某个数据帧定时器，超时后产生 `DATA_TIMEOUT`
- `stop_timer(nr)`：停止对应定时器
- `start_ack_timer(ms)` / `stop_ack_timer()`：管理 ACK 搭载定时器，超时后产生 `ACK_TIMEOUT`

必须理解的差别：

- `start_timer()` 的计时起点不是“调用瞬间”，而是**当前已进入物理层发送队列的数据发送完毕之后**
- `start_ack_timer()` 从**当前时刻**开始计时
- 在 ACK 定时器未超时前再次调用 `start_ack_timer()`，仍按原先设置的超时时刻触发

对选择重传实现来说，这部分直接关系到：

- 每个未确认数据帧的独立超时重传
- ACK 单独发送还是搭载发送
- 定时器参数如何影响利用率

### 5.3 日志与调试输出

```c
extern void lprintf(char *fmt, ...);
extern void dbg_event(char *fmt, ...);
extern void dbg_frame(char *fmt, ...);
extern void dbg_warning(char *fmt, ...);
char *station_name(void);
```

说明：

- `lprintf()` 会在输出前自动加上当前时间坐标，并同步写入日志文件。
- `dbg_event()` / `dbg_frame()` / `dbg_warning()` 最终都会调用 `lprintf()`。
- `station_name()` 返回当前站点名，值为 `"A"` 或 `"B"`。

`debug_mask` 控制位：

- bit0：事件信息
- bit1：帧收发信息
- bit2：警告信息

建议至少输出以下调试信息：

- 数据帧发送 / 接收
- ACK / NAK 或等价控制信息发送 / 接收
- CRC 错误
- 缓存窗口滑动
- 超时重传
- ACK 定时器触发

## 6. 命令行选项

帮助信息摘要如下：

```text
./datalink <options> <station-name>

-?, --help
-u, --utopia
-f, --flood
-i, --ibib
-n, --nolog
-d, --debug=<0-7>
-p, --port=<port#>
-b, --ber=<ber>
-l, --log=<filename>
-t, --ttl=<seconds>
```

实验中最常用的选项：

- `-u` / `--utopia`
  - 将本站接收方向信道误码率设为 `0`
  - 用于无误码环境测试
- `-f` / `--flood`
  - 网络层持续洪水式产生分组
  - 用于压测吞吐和利用率
- `-i` / `--ibib`
  - 仅对站点 B 有意义
  - 将 B 的发送模式调成与默认模式相反的 IDLE/BUSY 周期
- `-n` / `--nolog`
  - 不生成日志文件
- `-d3` / `--debug=3`
  - 设置调试输出类别
- `-p10002` / `--port=10002`
  - 修改 TCP 端口
- `-b 1e-4` / `--ber=1e-4`
  - 指定接收方向误码率
- `-l file.log` / `--log=file.log`
  - 指定日志文件名
- `-t <seconds>` / `--ttl=<seconds>`
  - 设定运行时间

示例：

```text
datalink -fd3 -b 1e-4 A
datalink --flood --debug=3 --ber=1e-4 B
```

## 7. 常见错误与直接含义

以下错误在调试阶段最值得优先关注：

- `Station name must be 'A' or 'B'`
  - 启动时必须明确指定站点名
- `Network Layer: incorrect packet length`
- `Network Layer received a bad packet from data link layer`
  - 链路层交给网络层的分组内容或长度错误
- `**** WARNING: System too busy, sleep 15 ms, but be awakened 61 ms later`
  - 系统负载过高，可能影响定时与性能统计
- `Station A failed to bind TCP port`
  - A 端绑定端口失败，检查端口占用和防火墙
- `Station B failed to connect station A`
  - B 端连接失败，检查 A 是否已启动、端口是否一致
- `Physical Layer Sending Queue overflow`
  - 物理层发送队列溢出，说明发帧速度和流控设计有问题
- `recv_frame(): Receiving Queue is empty`
  - 未发生 `FRAME_RECEIVED` 就调用了 `recv_frame()`
- `get_packet(): Network layer is not ready for a new packet`
  - 未发生 `NETWORK_LAYER_READY` 就调用了 `get_packet()`
- `start_timer(): timer No. must be 0~128`
  - 定时器编号非法

## 8. 测试与验收判据

### 8.1 协议正常工作的判据

协议正常工作的标志不是“程序没崩”，而是：

- 发送端持续有数据时，接收端持续成功收到正确分组
- `put_packet()` 的统计报告会隔一段时间持续出现
- 报告长期不再出现，通常意味着协议已经死锁或失败

### 8.2 建议执行的测试

原实验中与当前实现直接相关的测试如下：

| 序号 | 启动命令 | 说明 |
| --- | --- | --- |
| 1 | `datalink au` / `datalink bu` | 无误码信道数据传输 |
| 2 | `datalink a` / `datalink b` | A 平缓发送，B 按默认 BUSY/IDLE 周期发送 |
| 3 | `datalink afu` / `datalink bfu` | 无误码信道，A/B 都洪水式产生分组 |
| 4 | `datalink af` / `datalink bf` | 有误码默认信道，A/B 都洪水式产生分组 |
| 5 | `datalink af --ber 1e-4` / `datalink bf --ber 1e-4` | 较高误码率下洪水发送 |

测试时至少记录：

- 运行是否能稳定持续 20 分钟
- 接收端线路利用率
- 是否出现死锁、重复重传、误收、长时间停顿

### 8.3 针对选择重传应重点观察的问题

- 单帧出错后是否只重传出错帧，而不是整窗回退
- 接收窗口是否允许缓存失序到达但校验正确的帧
- 累积 ACK / 搭载 ACK / 单独 ACK 的策略是否合理
- 窗口大小与数据/ACK 定时器是否导致明显利用率下降
- 双向同时发送时是否仍能稳定工作
