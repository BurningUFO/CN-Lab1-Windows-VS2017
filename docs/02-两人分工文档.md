# 计算机网络实验一：数据链路层滑动窗口协议设计与实现
## 两人分工文档

## 1. 文档说明

本文档用于明确本小组在《计算机网络实验一：数据链路层滑动窗口协议设计与实现》中的两人分工。  
本组采用如下协作方式：

- **成员A主责协议核心实现**
- **成员B主责测试验证与性能分析**
- **关键逻辑采用交叉评审**
- **实验代码、测试记录、文档材料统一放入 Git 仓库管理**

本文档只针对当前课程实验，强调分工清晰、边界明确、便于落地执行。

---

## 2. 分工原则

### 2.1 主责明确

本实验不是平均拆代码，而是采用“**主责 + 协作 + 验收**”方式推进。  
每项任务必须有一名主负责人，另一名成员承担协助、复核或验收职责，避免“都在做、但没人真正负责”。

### 2.2 核心文件单主责
由于核心协议实现集中在 `datalink.c`，该文件默认由 **成员A主改**。  
成员B不长期并行大改同一段核心逻辑，避免：

- Git 冲突频繁
- 状态变量和事件逻辑相互覆盖
- 协议行为和测试结论对不上

### 2.3 测试与实现并行推进
成员A完成一段核心逻辑后，成员B应尽快进行联调、日志观察和测试记录，不等到代码全部完成后再集中测试。

### 2.4 关键路径必须互相过目
以下内容不能只由一人单独定稿：

- 主循环事件处理
- 帧格式定义
- ACK/重传/定时器逻辑
- 数据上交条件
- 性能测试结论
- 实验报告中的关键分析结论

---

## 3. 成员A职责

成员A为 **协议核心实现主责**。

### 3.1 主负责内容
1. **协议总体设计落地**
   - 根据实验要求实现滑动窗口协议
   - 明确发送端、接收端、确认、重传的基本机制

2. **`datalink.c` 核心实现**
   - 主循环结构
   - 事件分支处理
   - 状态变量设计与维护
   - 发送窗口与接收窗口逻辑
   - 帧缓存与窗口推进

3. **帧格式与核心机制实现**
   - 帧字段设计
   - 发送端逻辑
   - 接收端逻辑
   - ACK 处理
   - 超时重传
   - 定时器启停逻辑

4. **根据测试反馈修复协议问题**
   - 结合日志和现象定位代码问题
   - 修改关键逻辑并保持主线可编译

### 3.2 协作职责
- 向成员B说明当前版本的协议实现思路
- 对关键状态变量、窗口变化、重传条件做必要解释
- 配合成员B理解测试现象与日志输出

### 3.3 复核对象
成员A需要复核成员B整理的以下内容：

- 测试现象是否描述准确
- 测试结论是否与代码行为一致
- 性能问题是否归因合理

---

## 4. 成员B职责

成员B为 **测试验证与性能分析主责**。

### 4.1 主负责内容
1. **编译运行与双站联调**
   - 工程编译检查
   - A/B 站启动验证
   - 基础运行命令整理
   - 日志文件观察与保存

2. **正确性测试**
   - 无误码场景测试
   - 有误码场景测试
   - 长时间运行测试
   - 异常现象记录

3. **性能测试与分析**
   - 线路利用率相关测试
   - 测试数据整理
   - 不同测试条件下的现象比较
   - 性能问题记录与分析整理

4. **文档与实验报告素材整理**
   - 联调记录
   - 性能测试记录
   - README 补充内容
   - 实验报告中的测试、现象、问题、结果分析素材整理

### 4.2 协作职责
- 对成员A完成的关键逻辑做测试视角检查
- 及时反馈“能否跑通、哪里异常、日志表现如何”
- 把代码问题转化为可复现的测试问题，而不是只说“程序有问题”

### 4.3 复核对象
成员B需要复核成员A完成的以下内容：

- 主循环关键逻辑是否可解释
- 帧收发行为是否与设计一致
- ACK/重传/定时器逻辑是否在日志中表现合理
- 版本修改后是否影响已有测试结果

---

## 5. 双方共同职责

以下内容由双方共同参与，但必须区分主次：

1. **协议方案确认**
   - 由成员A主讲实现方案
   - 成员B从测试和可验证性角度提出意见

2. **关键逻辑讨论**
   - 包括帧格式、ACK 机制、重传策略、窗口推进逻辑
   - 最终实现由成员A落地
   - 最终可验证性由成员B确认

3. **联调问题定位**
   - 成员B负责记录现象和日志
   - 成员A负责对应到代码逻辑

4. **实验报告收口**
   - 成员B主整理
   - 成员A负责补协议实现说明和技术解释

---

## 6. 边界说明

### 6.1 `datalink.c` 的边界
- `datalink.c` 由成员A主负责
- 成员B原则上不长期直接修改 `datalink.c` 的核心逻辑块
- 成员B对 `datalink.c` 的参与方式以：
  - 问题反馈
  - 修改建议
  - 伪代码说明
  - 关键逻辑复核
  为主

### 6.2 测试与记录的边界
- 成员B主负责测试执行、日志整理、测试记录和性能分析材料

| 协议总体设计落地     | 成员A  | 成员B     |
| 帧格式设计           | 成员A  | 成员B     |
| 发送端逻辑           | 成员A  | 成员B     |
| 接收端逻辑           | 成员A  | 成员B     |
| ACK/重传/定时器机制  | 成员A  | 成员B     |

| 日志分析             | 成员B  | 成员A     |

## 8. 时间阶段划分

### 阶段一：工程与环境跑通
- 成员A：确认代码框架与实现入口
- 成员B：确认工程可编译、A/B 站可启动、日志可观察

### 阶段二：核心协议实现
- 成员A：完成主循环、帧处理、窗口、ACK、重传、定时器等核心逻辑
- 成员B：同步做基础联调和现象记录

### 阶段三：正确性与稳定性验证
- 成员A：根据测试结果修复问题
- 成员B：完成无误码、有误码、长时间运行测试

### 阶段四：性能测试与报告整理
- 成员B：主负责性能测试、结果整理、报告素材汇总
- 成员A：补充协议实现说明，协助解释性能结果和问题原因

## 9. 互相检查与验收方式

本组不采用复杂评审流程，只保留必要检查。

### 9.1 成员A完成后，成员B验收
成员B主要检查：

- 是否能编译运行
- A/B 站是否能正常启动
- 日志是否能反映协议行为
- 是否出现明显错误、卡死、异常退出
- 当前版本是否达到本阶段测试目标

### 9.2 成员B整理后，成员A验收
成员A主要检查：

- 测试记录是否准确
- 现象与日志是否能对应到协议逻辑
- 问题归因是否合理
- 性能分析是否与当前实现一致

### 9.3 关键逻辑共同确认
以下内容必须双方都看过再作为阶段性结果保留：

- 帧格式最终版本
- 主循环关键事件处理
- ACK/重传/定时器机制
- 主要测试结论
- 报告中的核心分析表述

## 10. 动态调整机制

如果后期出现以下情况：

- 分工明显不均衡
- 成员A实现压力过大
- 成员B测试工作已阶段性完成
- 实现与测试耦合度过高，单人推进困难

则按以下方式动态调整：

### 10.1 可调整内容
- 成员B短期参与部分辅助编码，但不直接长期并行重写 `datalink.c` 主体
- 成员A可把部分非核心整理工作交由成员B主做
- 双方可临时针对某一问题进行结对调试

### 10.2 调整原则
- 不改变“成员A主核心实现、成员B主测试分析”的总体结构
- 调整必须具体到事项，不采用“后面再看”式模糊分工
- 调整后仍需明确：
  - 谁主负责
  - 谁复核
  - 谁最终交付

### 10.3 不建议的调整方式
- 两人长期同时大改 `datalink.c`
- 所有事项都改成“共同负责”
- 只调整工作量，不调整验收责任

---

## 11. 最终执行口径

本组采用 **“成员A主核心实现，成员B主测试分析，关键逻辑交叉评审”** 的分工模式。  
执行时以以下原则为准：

1. `datalink.c` 由成员A主负责  
2. 测试、日志、性能分析由成员B主负责  
3. 关键逻辑与关键结论必须互相确认  
4. 避免长期无边界地同时修改同一核心逻辑  
5. 所有阶段都保留可用于实验报告的材料  

本分工文档用于保证本组协作边界清晰、推进顺畅、最终提交材料完整。
