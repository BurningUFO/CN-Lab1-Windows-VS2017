# 计算机网络实验一：数据链路层滑动窗口协议设计与实现
## 双人小组协作方案（精简版）

## 1. 文档目的

本文档用于约定本小组在《计算机网络实验一》中的实际协作方式。  
本组为 **2 人学生小组**，实验方案选择 **选择重传协议（Selective Repeat, SR）**，开发方式为 **Git 仓库协作**。

本方案只解决本实验中最实际的问题，不采用复杂项目流程，目标是：

- 分工明确
- 核心文件少冲突
- 代码、测试、报告能同步推进
- 最终材料便于整理和提交

---

## 2. 小组分工

### 成员A：核心协议实现主责
主要负责：

- `datalink.c` 中的核心协议实现
- SR 协议中的发送窗口、接收窗口、缓存、确认、重传、定时器逻辑
- 关键 bug 修复
- 对核心实现思路做必要说明

### 成员B：测试验证与性能分析主责
主要负责：

- 程序启动、双站联调、日志观察
- 正确性测试、长时间运行测试、性能测试
- 测试现象记录与整理
- 实验报告中的测试结果、问题分析、性能分析部分
- 对成员A完成的关键协议逻辑做交叉检查

### 协作方式
本组采用：

- **一人主写核心实现**
- **一人主做测试和分析**
- **关键逻辑互相过目**

这样分工的原因是：

- 本实验核心逻辑集中在 `datalink.c`
- 选择重传协议状态较多，双人同时改核心文件容易冲突
- 对双人学生小组来说，这种方式最省沟通成本

---

## 3. 仓库使用原则

## 3.1 主线分支
仓库使用 `main` 作为主线分支，要求：

- 主线代码始终尽量保持可编译
- 不直接在 `main` 上长期开发
- 每个阶段完成后再合并回 `main`

## 3.2 临时分支
使用短分支进行开发，命名保持简单即可，例如：

- `feat/sr-core`
- `feat/recv-buffer`
- `fix/ack-bug`
- `test/basic-run`
- `docs/report`

说明：

- 成员A主要使用 `feat/*`、`fix/*`
- 成员B主要使用 `test/*`、`docs/*`

## 3.3 老师提供文件默认不改
以下文件原则上不作为主要修改对象：

- `protocol.c`
- `protocol.h`
- `crc32.c`
- `lprintf.c`
- `getopt.c`

本实验的主要实现集中在：

- `datalink.c`

如果确实需要修改老师提供文件，必须：

- 单独说明原因
- 单独提交
- 让另一人知道改了什么

---

## 4. 提交与合并约定

## 4.1 提交要求
每次提交尽量只表达一个清楚的改动，不要把多个无关内容混在一起。

提交前至少做到：

- 工程能编译
- 改动目的明确
- 提交信息能看懂

## 4.2 提交信息格式
建议格式：

```text
<type>: <说明>
```

常用类型：

- `feat`
- `fix`
- `test`
- `docs`
- `refactor`

示例：

```
feat: 完成SR接收窗口缓存逻辑
fix: 修正超时重传后的ACK处理
test: 记录无误码场景联调结果
docs: 更新实验报告测试提纲
```

## 4.3 合并要求

合并到 `main` 前，至少满足：

- 本地编译通过
- 对方知道这次改了什么
- 如果涉及 `datalink.c` 核心逻辑，至少做过一次基础联调

本组不做复杂 PR 流程，但要求：

- **核心实现改动要让对方看过**
- **测试结论写入文档前要让对方确认**

------

## 5. `datalink.c` 协作规则

这是本方案最重要的一部分。

## 5.1 单主责原则

`datalink.c` 默认由 **成员A主改**。
 成员B原则上不长期并行大改 `datalink.c`。

原因：

- SR 协议状态变量多
- 主循环、窗口、ACK、重传逻辑耦合强
- 双人同时改同一个文件，容易产生 Git 冲突和逻辑冲突

## 5.2 成员B如何参与 `datalink.c`

成员B对 `datalink.c` 的参与方式以“提出修改建议”为主，例如：

- 直接说明问题现象
- 标出怀疑的逻辑位置
- 写伪代码或文字建议
- 提交小 patch
- 在分支或评论中指出修改点

最终由成员A统一整合进 `datalink.c`。

## 5.3 什么情况必须互相确认

以下改动必须让对方先知道：

- 主循环结构变化
- 发送窗口 / 接收窗口推进逻辑变化
- ACK 处理方式变化
- 超时重传策略变化
- `put_packet()` 的交付条件变化
- 网络层启停条件变化

## 5.4 已发生冲突时的处理方法

如果 `datalink.c` 已经发生冲突，不直接拼代码，按下面处理：

1. 先说明双方各自改了什么逻辑
2. 先统一“最终希望的协议行为”
3. 再决定保留哪段实现
4. 由成员A完成最终整合
5. 整合后重新编译，并做一次基础联调

------

## 6. 测试与报告协作方式

## 6.1 成员B负责的测试输出

成员B负责持续整理以下内容：

- 启动命令
- 测试场景
- 是否通过
- 出现的现象
- 关键日志片段
- 性能测试结果
- 当前主要问题

这些记录后续直接作为实验报告素材。

## 6.2 成员A对测试结果的配合

成员A需要对成员B整理出的测试结果做两件事：

- 判断问题大致属于哪段协议逻辑
- 确认测试结论是否与当前实现一致

这样可以避免：

- 代码的人不看测试
- 测试的人不懂代码
- 最后报告里“现象”和“原因”对不上

## 6.3 建议保留的文档

建议仓库中至少保留：

```
docs/
  01-协作方案.md
  03-联调记录.md
  04-性能测试记录.md
  05-实验报告大纲.md
```

不需要太多文档，但上面这几份很有用。

## 7. 版本推进建议

本实验建议按以下几个阶段推进：

### M1：工程与环境跑通

目标：

- 工程可编译
- 双站能启动
- 仓库与基础文档建立完成

### M2：SR 基础框架完成

目标：

- 基本收发帧逻辑建立
- ACK 逻辑建立
- 发送窗口 / 接收窗口基本结构建立

### M3：SR 核心逻辑跑通

目标：

- 缓存失序帧
- 按序交付数据
- 单帧确认
- 超时重传

### M4：正确性与稳定性测试

目标：

- 无误码场景可运行
- 有误码场景不立即失败
- 长时间运行基本稳定

### M5：性能测试与报告整理

目标：

- 完成性能测试
- 整理利用率、问题分析、现象解释
- 完成实验报告

------

## 8. 主要风险与应对

### 风险1：两人同时改 `datalink.c`

应对：

- 保持单主责
- 另一人以建议、测试、评审为主
- 关键修改提前说明

### 风险2：SR 逻辑复杂，越写越乱

应对：

- 按小功能推进
- 每次只解决一个关键问题
- 核心变量和状态变化要能说清楚

### 风险3：代码能跑，但测试材料不足

应对：

- 从一开始就记录测试现象
- 失败场景也保留
- 日志、命令、测试结果一起存档

### 风险4：报告阶段才开始补记录

应对：

- 成员B持续维护测试记录
- 每轮联调后立刻补一小段记录
- 不等到最后统一回忆

------

## 9. 本组执行口径

本组采用 **“一人主实现，一人主测试，关键逻辑交叉确认”** 的简单协作方式。
 实际执行时，只坚持下面几条即可：

1. `datalink.c` 主要由成员A维护
2. 成员B尽早测试、尽早暴露问题
3. 核心逻辑改动必须同步
4. 关键测试结果必须留记录
5. 主线始终保持可编译、可继续推进

本方案不追求形式化，只要求真正适合本组执行。
